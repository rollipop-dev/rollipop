title: MCP Server

The dev server embeds an [MCP (Model Context Protocol)](https://modelcontextprotocol.io) server at `/mcp`, allowing LLM agents like Claude Code to interact with the bundler through standard MCP tooling.

## Setup

Add the MCP server to your project's `.mcp.json`:

```json
{
  "mcpServers": {
    "rollipop": {
      "type": "http",
      "url": "http://localhost:8081/mcp"
    }
  }
}
```

Make sure the dev server is running before starting the LLM agent.

## Available Tools

### `reset_cache`

Clears the entire build cache. The bundler will rebuild from scratch on next file change.

### `get_build_events`

Subscribes to bundler events for a specified duration and returns all collected events.

| Parameter  | Type   | Default | Description                                     |
| ---------- | ------ | ------- | ----------------------------------------------- |
| `duration` | number | 10000   | How long to listen in milliseconds (1000-60000) |

Returns a JSON array of events collected during the listening period. Event types include:

- `bundle_build_started` / `bundle_build_done` / `bundle_build_failed` — build lifecycle
- `watch_change` — file change detected
- `client_log` — app console output
- `device_connected` / `device_disconnected` — device lifecycle
- `cache_reset` — cache cleared via control API

Bundler-scoped events include `bundlerId` in their JSON payload. This includes build lifecycle events, `watch_change`, and `client_log` when the HMR client is associated with a bundler instance. Device, server, and cache events omit `bundlerId`.

See [SSE](/docs/features/sse) for the full event type reference.

## Polling bundler state

In addition to the MCP tools above, the dev server exposes a plain HTTP endpoint that returns the current lifecycle state of a specific bundler as JSON:

```
GET /bundlers/<id>/status
```

The `<id>` is the same value carried as `bundlerId` by bundler-scoped SSE/MCP events. The response is a point-in-time snapshot:

```json
{ "id": "<id>", "status": "build-done" }
```

| `status`       | Meaning                                                      |
| -------------- | ------------------------------------------------------------ |
| `idle`         | Bundler instance created but the first build has not started |
| `building`     | A build is currently in progress                             |
| `build-done`   | The most recent build completed successfully                 |
| `build-failed` | The most recent build (or HMR update) failed                 |

Unknown ids return `404` with `{ "error": "not found" }`. The bare `/status` path is unaffected — it continues to return `packager-status:running` from the React Native community middleware.

Agents can use this as a cheap pull-based alternative to `get_build_events` when they only need the latest state of a known bundler rather than the full event stream — for example, polling until a rebuild settles after an edit.

## LLM Workflow

An LLM agent can use MCP tools to interact with the bundler:

1. Call `get_build_events` to observe bundler state
2. Make code changes
3. Call `get_build_events` again to wait for `bundle_build_done` or `bundle_build_failed` — or poll `GET /bundlers/<id>/status` for a single known bundler
4. If failed, read the error and fix the code
5. If needed, call `reset_cache` to clear stale cache
